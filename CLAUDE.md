# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is a **.NET 10 ASP.NET Core app (`Microsoft.NET.Sdk.Web`) hosting Blazor Server plus a background worker**, designed to run as a **Windows Service** (`builder.Host.UseWindowsService()`). It started life as a Worker Service template and was converted: the same process now serves an interactive Blazor Server UI *and* runs `Worker` as a hosted background service. The actual backup functionality has not been implemented yet — `Worker.cs` still just logs a heartbeat — so most work here is greenfield.

## Commands

Run from the repo root (`C:\Projects\BackupService`). The solution is `BackupService.slnx` (the newer XML solution format); the project lives in `BackupService/`.

```powershell
dotnet build                          # build the solution
dotnet run --project BackupService    # run web host + worker (Ctrl+C to stop)
dotnet watch --project BackupService run   # run with hot reload
```

Running interactively (console / `dotnet run`) starts Kestrel and serves the Blazor UI. `UseWindowsService()` only changes the host lifetime when the process is actually started by the Windows SCM, so local console runs are unaffected. To install as a Windows Service, publish then run `BackupService.exe --install` from an elevated prompt (see the install commands below).

**HTTP only — deployed on `http://localhost:55000`, dev on `http://localhost:5080`.** There is no HTTPS (the admin UI is local-only). The port lives in **environment-specific** config (not base `appsettings.json`): `appsettings.Production.json` → `Kestrel:Endpoints:Http:Url` = 55000 (the deployed service runs in Production), `appsettings.Development.json` → 5080. Base `appsettings.json` intentionally has **no** `Kestrel:Endpoints` — otherwise Visual Studio reads it as the default and launches the browser on 55000 even in debug. The dev launch profile (`launchSettings.json`) sets `ASPNETCORE_ENVIRONMENT=Development` (web-standard) and `applicationUrl=5080`. Note a `Kestrel:Endpoints` value takes precedence over `ASPNETCORE_URLS`/`applicationUrl`, so the per-environment port is set via the config layer.

**Install as a Windows Service:** run the published exe elevated — `BackupService.exe --install` (registers autostart + LocalSystem, replaces an existing service, and starts it) or `BackupService.exe --uninstall` (stops + removes it). These commands run and exit without starting the web host; see `Hosting/WindowsServiceInstaller.cs` (shells out to `sc.exe`). Must be run from the published exe, not `dotnet …dll`.

**Deploy via publish profile:** `Properties/PublishProfiles/Install.pubxml` publishes to `C:\Tools\BackupService` and (re)installs the service — `dotnet publish -p:PublishProfile=Install`. It stops the running service before copying (so files aren't locked) and runs `--install` afterward. **Run it from an elevated prompt**; otherwise the deploy still succeeds but the install step only warns (it needs admin). The ASP.NET Core dev HTTPS certificate / "trust the self-signed cert" prompt is irrelevant here and does not apply to the service. If the UI ever needs network/HTTPS access, configure a real cert under `Kestrel:Endpoints` and add `CookieSecurePolicy.Always` + `UseHttpsRedirection()`.

Note when running the built dll directly for testing: set the working directory to the output folder, otherwise the content root is the current directory and `appsettings.json` (hence the Kestrel binding) won't be found. The real service is unaffected because `UseWindowsService()` sets the content root to the exe's directory.

Tests live in the sibling **`BackupService.UnitTests`** project (in the solution). Stack: **NUnit** + **Moq** + **FluentAssertions**. Note FluentAssertions is pinned to **7.x** (the last Apache-2.0/free line — 8.x requires a paid commercial license); don't bump to 8.x without a license.

```powershell
dotnet test                                                   # run all tests
dotnet test --filter "FullyQualifiedName~TestClassName.TestMethodName"   # single test
```

Testing services that use the DbContext factory: mock `IDatabaseContextFactory` with Moq and have `CreateDbContext()` return a `BackupDbContext` over a **shared in-memory SQLite** connection (`new SqliteConnection("DataSource=:memory:")` kept open for the test, `EnsureCreated()` for schema) — see `Authentication/AdminCredentialServiceTests.cs` for the pattern. Use `NullLogger<T>.Instance` for `ILogger` dependencies.

### Database / EF Core migrations

Migrations are kept under `BackupService/Database/Migrations`. Always pass `--output-dir Database/Migrations` when adding one so it lands there rather than the default `Migrations/` folder. The `dotnet-ef` tool version must match the EF Core packages (currently 10.0.9).

```powershell
dotnet ef migrations add <Name> --project BackupService --output-dir Database/Migrations
dotnet ef database update --project BackupService     # apply migrations manually
dotnet ef migrations remove --project BackupService   # undo the last (unapplied) migration
```

The app also calls `db.Database.Migrate()` on startup, so pending migrations apply automatically when it runs. The SQLite file (`backupservice.db`, plus `-wal`/`-shm`) is generated at the content root and is not source.

## Architecture

- `Program.cs` — host bootstrap via `WebApplication.CreateBuilder`. Calls `UseWindowsService()`, registers the DB factory, cookie auth + authorization, Razor Components with interactive server rendering (`AddRazorComponents().AddInteractiveServerComponents()`), registers the backup scheduler as the hosted service, then maps the Blazor endpoint (`MapRazorComponents<App>().AddInteractiveServerRenderMode()`) and a `/logout` POST endpoint. The middleware pipeline is `UseAuthentication` → `UseAuthorization` → `UseAntiforgery` → `MapStaticAssets` → endpoints. Startup also runs `db.Database.Migrate()` and `IAdminCredentialService.EnsureSeededAsync()`. **Static assets use .NET fingerprinting:** `app.MapStaticAssets()` (not `UseStaticFiles`) serves `wwwroot`, and `App.razor` links the stylesheet via `@Assets["app.css"]` so the URL carries a content hash — CSS/JS edits bust the browser cache automatically (avoids the stale-CSS problem where new markup loads but old styles persist).
- **Backup execution is driven by the scheduler, not a `Worker`.** (`Worker.cs` — the old idle heartbeat placeholder — has been removed; `BackupSchedulerService` is the `BackgroundService` now.) Backup logic belongs in the per-type handlers in `Scheduling/` (see below), not inline in the host.
- `Database/` — **all** database code lives here: `BackupDbContext` (EF Core, SQLite), the entity models, the `Migrations/` subfolder, `BackupDatabaseLocation`, and the `DatabaseContextFactory`. Keep new entities and DbContext changes in this folder. Core domain: a `Profile` has one-to-many `FolderPair` records (cascade delete); the `FolderPair` carries status enums (`FolderPairStatus`, `FolderPairLastRunStatus`). The second profile type, **`ProfileType.InstantSync`**, has its own one-to-many child, **`InstantSyncItem`** (cascade delete) — each with `Name`, `SourceFolder`, `TargetFolder`, an `int DebounceMilliseconds`, `IncludeSubFolders` and `AllowDeletions` (no status/overwrite columns — instant sync is source-authoritative and has no discrete run). The third profile type, **`ProfileType.ArchiveSync`**, has its own one-to-many child, **`ArchiveSyncItem`** (cascade delete) — each with `Name`, `SourceFolder`, `TargetFolder`, a `FileName` (the base archive name a timestamp is appended to), `IncludeSubFolders`, an `ArchiveRetentionMode RetentionMode` (`KeepLastN`/`GrandfatherFatherSon`), an `int RetentionCount` (N — total for keep-last-N, per-level for GFS), an `int MaxLevels` (GFS level count; 1 otherwise) and an `int RunCount` (monotonic count of archives created — the only mutable run-state, drives the GFS promotion cadence; no Status/LastRunStatus columns). ArchiveSync is **scheduled** (uses `Profile.Schedule`/the scheduler, like FolderPair — not watcher-driven). A profile uses the child collection matching its `Type` (`FolderPairs`, `InstantSyncItems` or `ArchiveSyncItems`); the others stay empty. **`FolderPair` and `ArchiveSyncItem` each own a one-to-many `Filters` collection** (`FolderPairFilter` / `ArchiveSyncFilter`, cascade-deleted via the default required-FK convention) — include/exclude rules each with a `FilterDirection Direction` (Include/Exclude), a `FilterKind Kind` (File/Folder/**Path**) and a `Pattern`. File/Folder are **name-only wildcards** (matched in every sub-folder); **`Path` (exclude only)** is a literal path *relative to the source root*, matched against the exact location (that file, or that folder and its whole subtree) — not by name. Empty includes = back up everything; see `BackupFilter` in `FileSystem/`. **A profile's own run status (`ProfileStatus` — Idle/Running/Error) is deliberately NOT persisted** — it describes the current run and is tracked in memory by `IProfileStatusService` (see `Profiles/`); there is no `Profile.Status` column. Each `FolderPair` also has an `OverwriteBehaviour` enum (how to treat an existing/newer file at the destination). `Profile` has a `bool Enabled` (on/off switch) that **defaults to true** via a store default — `BackupDbContext.OnModelCreating` configures `Property(p => p.Enabled).HasDefaultValue(true)` so the migration backfills existing rows to enabled; the C# initializer is also `= true`. (EF Core 7+ then treats `true` as the sentinel, so a `false` value is still persisted — disabling a profile works.) Other tables: `AdminCredential`, `AuthenticationHistory`, `BackupRecord`, and `LogRetentionSettings` (a **single-row** settings table — `AuthenticationLogRetentionDays` default 7, `OperationLogRetentionDays` default 30 — seeded lazily by `ILogRetentionService`; drives the log-purge below). There is also an operation-logging pair: `OperationLog` (header — unbounded `Name`, `TimestampUtc`, and an `OperationLogLevel` enum — Info/Warning/Error/Debug — the **severity of the operation as a whole**) has one-to-many `OperationLogDetail` rows (cascade delete; unbounded `Message`, `TimestampUtc`, an `int Sequence` for ordering lines within a log, and **its own `OperationLogLevel Level`** — each line carries its severity). **The header `Level` is derived from the detail lines, not set independently:** it is the most severe of its lines' levels — all Info/Debug → Info, any Warning → Warning, any Error → Error (Debug counts as Info for this) — maintained by `OperationLogger` as lines are appended (a header only ever rises; see `Logging/`). A detail-less log keeps the level it was created with. `OperationLog` also has a **nullable** `int? ProfileId` FK to `Profile` (null for logs not tied to a profile) configured in `OnModelCreating` with `OnDelete(DeleteBehavior.Cascade)` — deleting a profile cascade-deletes its logs (cascade is the non-default behaviour for a nullable FK, so it's set explicitly). Note the profile *deletion* log is deliberately left unassociated (`ProfileId` null) so it survives the delete. Written via the `Logging/` factory; read/displayed by the Logs sidebar section.
  - **Factory pattern — the `DbContext` is NOT registered in DI.** Instead `IDatabaseContextFactory` (impl `DatabaseContextFactory`) is registered as a singleton. Code that needs the database injects `IDatabaseContextFactory` and calls `CreateDbContext()`, owning and disposing a short-lived context per unit of work (`using var db = factory.CreateDbContext();`). Do **not** add `AddDbContext<BackupDbContext>` back. `DatabaseContextFactory` also implements `IDesignTimeDbContextFactory<BackupDbContext>` so `dotnet ef` can build a context at design time without the context being in DI.
  - **SQLite caveat:** SQLite cannot `ORDER BY` (or compare) a `DateTimeOffset` column. For "newest first" ordering use the autoincrement `Id` (monotonic with insertion) rather than a timestamp column — see `AuthenticationHistoryService.GetPageAsync`. `DateTimeOffset` columns are fine to store and `SELECT`/display.
  - **Single database, location depends on hosting** (`BackupDatabaseLocation.GetConnectionString()`): there is only ever one SQLite file. When running as a Windows Service it lives under ProgramData (`%ProgramData%\BackupService\backupservice.db`) so it's shared machine-wide regardless of the service account; when run interactively (debugging/console) it sits next to the exe (`AppContext.BaseDirectory`). The decision uses `WindowsServiceHelpers.IsWindowsService()`. There is no connection string in `appsettings.json` — it is computed in code.
- `Components/` — the Blazor Server UI. `App.razor` is the root HTML document (loads `_framework/blazor.web.js` and the theme `app.css`), `Routes.razor` wires the router via `AuthorizeRouteView` (unauthenticated → `RedirectToLogin`), and routable pages live in `Components/Pages/` (`@page` directives). `_Imports.razor` holds shared `@using`s for the component tree. Two layouts: `Components/Layout/MainLayout.razor` (default — top bar with signed-in user + Log out, gated by `<AuthorizeView>`, content in `.page`) and `Components/Layout/AuthLayout.razor` (full-screen centered, used by `Login` via `@layout AuthLayout`).
- **Layout shell:** `MainLayout` wraps the top bar + `<main class="app-main">` in `.app-shell` (a flex column at `height: 100vh`), so `.app-main` fills the viewport below the title bar. Pages that want a centred, max-width look opt into a `.page` wrapper; pages that need full height/bleed use `.app-main` directly.
- **Workspace pages (sidebar + content):** both the home **`BackupServicePage`** (route `/`, title "Backup Service") and the **Settings** page use the shared `.workspace` / `.sidebar` / `.sidebar-item` / `.workspace-content` CSS: a left-docked navigation sidebar (a self-contained control owning its items, two-way `@bind-SelectedKey`) plus a content area that swaps the selected section's control. `BackupServicePage` has `BackupServiceSidebar` (items: Dashboard → `Dashboard.razor`, Backup Profiles → `Profiles.razor`, Logs → `Logs.razor`); Settings has `SettingsSidepanel` (items: Authentication → `AuthenticationPanel`, Logging → `LoggingPanel` (log retention settings, `Components/Pages/Logging/`)). Add a section = one sidebar `NavItem` (key/label/icon) + a matching `case` arm + a panel control. The **Logs** section is a paged table of `OperationLog` headers (newest first, via `IOperationLogService`) — each row shows the header's level as a coloured `.log-level` pill, the associated **Profile** name (or a muted `—` when not tied to one), the log name, and timestamp, plus a `+`/`−` button that expands a **terminal-style view** of that log's `OperationLogDetail` lines — monospace `[{level}] [{timestamp}] {message}` rows (no wrap, like a real terminal; long messages scroll horizontally), each line tinted by its own `Level` (Warning → amber, Error → red, Info/Debug → default), lazy-loaded on first expand and cached. A `message` may itself contain **CRLFs** (a consolidated single row whose lines are split by `\r\n` — e.g. the ArchiveSync "Archived {n} file(s):" list); `.log-terminal-line` is `white-space: pre` with `min-height` (not a fixed height) so such a row grows to multiple visual lines. Above the terminal view sits a per-log `.log-detail-filter` bar — a free-text line filter and a level `Dropdown` (`OperationLogLevel?`, null = "All levels") — that narrows the displayed lines **client-side** from the cached set (the free-text input is wrapped in `.filter-clearable`, which shows a clear `&times;` button when it has content — the same wrapper/`.filter-clear` pattern used by the header name filter) (each expanded log keeps its own filter state, keyed by log id, in `_detailFilters`/`_detailLevels`; `FilteredDetails(logId)` applies a case-insensitive `Message` `Contains` and exact `Level` match before feeding `Virtualize`). Because a big backup can write thousands of lines, the lines are rendered with `<Virtualize>` (only visible rows are in the DOM) inside a `.log-terminal` box capped at ~20 lines (`max-height` = 20 × the 20px `.log-terminal-line` height, matched by `LineHeight`/`ItemSize` in `Logs.razor.cs`) that scrolls vertically beyond that. The detail cache is a `List<OperationLogDetail>` so it can feed `Virtualize.Items`. The expand control only shows when the log actually has detail lines — `GetPageAsync` populates a non-persisted `OperationLog.DetailCount` ([NotMapped], via a grouped count query for the page) and the grid hides the `+` when it's 0, so self-describing logs (message-in-name, no sub-entries) render as a plain row. Above the table a `.filter-bar` has a free-text name filter (live, `@oninput`), an **Include Messages** checkbox, a level `Dropdown` (`OperationLogLevel?`, null = "All levels"), and a profile `Dropdown` (`int?` profile id, null = "All profiles"; options come from `IProfileService.GetSummariesAsync`). `GetPageAsync(page, size, filter, includeMessages, level, profileId)` does the matching server-side — name (and detail `Message` too when the box is ticked) via `EF.Functions.Like` (LIKE is case-insensitive for ASCII in SQLite), an exact `Level` match, and an exact `ProfileId` match; all filters combine with AND, and each returned log has its `Profile` navigation `Include`d for the column.
- **Naming caveat:** never name a type or page folder `BackupService` — it equals the root namespace and shadows it (breaks `@using BackupService...`). The home page folder/class is `BackupServicePage` for this reason.
- **Interactivity:** pages are static SSR by default. Components needing interactivity opt in with `@rendermode InteractiveServer` (infrastructure already wired in `Program.cs`). The Settings page is the first such page (the `SettingsSidepanel` nav updates the shown section via `@bind-SelectedKey`). Keep the login page static SSR (it must write the auth cookie on its form POST).
- **Theme/styling:** a single global dark theme in `wwwroot/app.css`, driven by CSS custom properties under `:root` (`--bg`, `--surface`, `--accent`, …) — re-skin by editing those variables. Shared classes: `.topbar`, `.page`, `.card`, `.auth-shell`/`.auth-card`, `.btn-secondary`, `.alert-error`. No CSS framework (no Bootstrap). Note: `wwwroot` is served via `app.MapStaticAssets()` (the Web SDK static-web-assets system with fingerprinting) — live from source in Development and copied into `wwwroot` on `publish`. Reference the stylesheet via `@Assets["app.css"]` (already done in `App.razor`) so the served URL is content-hashed; a plain `href="app.css"` caches indefinitely and shows stale styles after edits. (A raw `dotnet bin\...dll` run in Production still can't serve `/app.css` because the assets aren't copied to `bin/` on a plain `build` — a test artifact only, not a real-deployment issue.)

It opens `FolderPairEditDialog` (in `Components/Dialogs/`) to add/edit a pair. The `FolderPairModel` view model lives here alongside `FolderPairControl`. `InstantSyncControl` (+ its `InstantSyncItemModel`) is the InstantSync counterpart — same list/Add/Validate shape, opening `InstantSyncEditDialog`. `ArchiveSyncControl` (+ its `ArchiveSyncItemModel`) is the ArchiveSync counterpart, opening `ArchiveSyncEditDialog`. **The `FolderPairModel`/`ArchiveSyncItemModel` view models also carry `List<FilterEntryModel> Includes`/`Excludes`** (the include/exclude rules), which `Clone(...)` deep-copies (via `FilterEntryModelExtensions.Clone`) so an edit can be cancelled. `TabBar` — a small horizontal tab strip (the `SettingsSidepanel` section-switcher pattern, laid out horizontally): `Items` (key/label `TabItem` records) + two-way `@bind-SelectedKey`; the parent switches the shown content. Used by the tabbed `FolderPairEditDialog`/`ArchiveSyncEditDialog`. `FilterListEditor` — edits one tab of include/exclude entries (reused for both the Includes and Excludes tabs of both dialogs): a type radio (Include ⇒ "Specific file"/"Wildcard pattern"; Exclude ⇒ "File"/"Folder"/**"Path"**), a pattern box + Add, and a list with a derived type tag + remove. Takes `Entries` (this tab) + `OtherEntries` (other tab, for cross-tab clash) + `Direction` + an optional **`SourceFolder`** (the parent's source). Browse is disabled until a source folder is set. Adds are gated by `FilterValidation`. The schedule is a profile-level field collected by `ProfileDialog` (which hosts the `ScheduleDialog`), not by `FolderPairEditDialog`.
- `Scheduling/` — `ScheduleMode` + `ScheduleDefinition` (the user-friendly schedule selection). `ToCron()` makes a standard 5-field cron string (stored in `Profile.Schedule`), `ToHumanReadable()` makes the label. **The raw cron is never shown to the user — always humanize.** `FromCron(cron)` parses the stored cron back into a definition (used to pre-fill the edit dialog's schedule builder) and `Describe(cron)` is the always-safe display helper ("Not scheduled" / friendly label / raw cron only as a last-resort fallback) used by the grid. Cron is validated/parsed with the **Cronos** package (also used for next-run times by the scheduler below).
  - **The scheduler that actually fires profiles** also lives here. `BackupSchedulerService` is a singleton registered in **three roles** — itself, `IBackupScheduler`, and the hosted `BackgroundService` (one shared instance via `AddSingleton(...)` + `AddSingleton<IBackupScheduler>(sp => sp.GetRequiredService<...>())` + `AddHostedService(sp => ...)`). On startup it loads every enabled profile that has a `Schedule` and registers it; a loop computes each profile's next run with Cronos (`CronExpression.GetNextOccurrence(DateTimeOffset.Now, TimeZoneInfo.Local)` — **local wall-clock, DST-correct**), waits until the soonest (capped at 1 min, and woken early on change via a `SemaphoreSlim`), then fires every due profile. **Jobs run concurrently** (each due profile launched on its own `Task.Run`, so a slow backup never blocks others) and a profile is **never run twice at once** — before spawning a thread the scheduler skips a profile whose run is still in progress (`IProfileStatusService.IsRunning`), and `BackupRunner` re-checks atomically via `TryBeginRun` (the authority). It reads profiles via `IDatabaseContextFactory` directly (not `IProfileService`) to avoid a DI cycle. The `IBackupScheduler` API is a single `SyncAsync(profileId)` that re-reads the profile and schedules it (enabled + parseable cron) or unschedules it (missing/disabled/blank/invalid) — `ProfileService` calls it after **every** mutation (`CreateAsync`/`UpdateAsync`/`SetEnabledAsync`/`DeleteAsync`) so edits take effect immediately. (`GetNextOccurrence(cron, from, tz)` is a pure static helper, extracted for tests.)
  - **What runs when a schedule fires is dispatched by `ProfileType`.** `IBackupRunner`/`BackupRunner` (singleton) is the dispatcher: `RunAsync(profileId, manual = false)` loads the profile **with its `FolderPairs`**, **skips the run if the profile is locked** (`IProfileStatusService.IsLocked` — an admin has it open in an edit/delete dialog), otherwise sets the in-memory status to `Running`, runs the handler, **stamps `DateLastRun` (the only DB write — status is no longer persisted) and *then* flips the status to `Idle`** (order matters: the grid reloads on the status change, so the timestamp must be saved first or the grid shows the previous run's `DateLastRun`). On failure the handler has already set `Error`; the runner stamps `DateLastRun` and re-asserts `Error` so the grid's status-triggered reload also picks up the fresh timestamp. It does **not** write its own operation log — **the handler owns the single per-run log** (see next). It calls the `IProfileTypeHandler` whose `Type` matches `profile.Type` (handlers injected as `IEnumerable<IProfileTypeHandler>`, indexed into a dictionary); an unmapped type is the one case the runner itself logs (a "Scheduled backup failed" error log) since no handler runs. On a handler exception the runner just records the failure via `ILogger` (the handler already set `Error` status and logged — see next). **A run is either scheduled or `manual`** — `manual: true` is a "Run now" from the Profiles grid (see `Profiles/`), routed through the *same* `RunAsync` so it goes through the identical status/locking/single-instance path; the only difference is the run's operation log is **prefixed `[Manual] `**. `IProfileTypeHandler` is the per-type seam (`ProfileType Type { get; }` + `HandleAsync(Profile, manual, ct)`); `FolderPairHandler` (`Type => ProfileType.FolderPair`) and `InstantSyncHandler` (`Type => ProfileType.InstantSync` — a manual full reconcile, see the InstantSync bullet below) are the two so far. (`BackupRunner` `Include`s both `FolderPairs` and `InstantSyncItems` so whichever handler runs has its items.) It creates **exactly one** `OperationLog` per run: the header starts as `"{prefix}{Type.GetDescription()} Handler called with {n} folder pair(s)."` (e.g. "Folder Pairs Handler called with 2 folder pair(s).", or "[Manual] Folder Pairs Handler …" for a manual run), and a `finally` rewrites the header (via `SetSummaryAsync`) to `"{prefix}… ran successfully in {duration} — {c} copied, {u} updated, {d} deleted"` (Info) or `"{prefix}… completed with {n} error(s) in {duration} — …"` (Error when any file/folder op failed) or `"{prefix}… failed in {duration}"` (Error for a *catastrophic* failure outside the per-pair loop). The "{ProfileType description} Handler" prefix is derived from the enum's `[Description]`, and the log already carries the profile via its FK. **The actual file work is `IFolderPairSynchronizer`/`FolderPairSynchronizer`** (singleton; see `FileSystem/`/`Scheduling/`): for each pair the handler sets `FolderPair.Status = Running` (DB), appends a `"Folder pair '{Name}': {src} -> {tgt}"` line, calls `synchronizer.SyncAsync(pair, log, ct)`, aggregates the returned `BackupResult` (Copied/Updated/Deleted/Errors), then persists `LastRunStatus = Success/Fail` + `Status = Idle`. A per-pair unexpected exception is logged and the run continues with the other pairs; only a catastrophic failure sets the in-memory `ProfileStatus.Error` and re-throws (per-file errors stay logged-but-non-fatal — the Error detail lines escalate the header to Error and the pair's `LastRunStatus` is `Fail`, but the run completes and the grid status stays Idle). **Add a profile type = one `IProfileTypeHandler` impl + its `AddSingleton<IProfileTypeHandler, …>()` registration** — no scheduler/runner change; each handler owns its single per-run log.
  - **The folder-pair sync engine** — `IFolderPairSynchronizer`/`FolderPairSynchronizer` (singleton) — walks the source tree **one folder at a time** and mirrors it into the target, logging **only operations actually performed** (no line for an unchanged file) to the run's `IOperationLogger` and returning a `BackupResult`. Per file, by `LastWriteTimeUtc` (**exact** UTC compare): destination missing or source newer → copy; equal → skip; destination newer → the pair's `OverwriteBehaviour` (`DoNotOverwriteNewer` skip / `AlwaysOverwrite` overwrite / `UpdateOnlyIfContentMatches` → if byte-equal, sync the destination's timestamp to the source's, else skip). `IncludeSubFolders` → depth-first recursion; `AllowDeletions` → **full mirror** (delete orphan target files *and* orphan sub-folders recursively). **Copies are crash-safe:** every copy/overwrite is written to a dot-prefixed `.{name}.tmp` in the target folder via `CopyThroughTempAsync`, then (on success) the original is deleted and the temp renamed onto it; on **any** error the temp is removed (best-effort) so a partial/temp file is never left behind. A file write/delete failure or a folder-access failure is logged as an **Error** line with the reason and counts toward `Errors` (the run continues). All filesystem access goes through **`FileSystem/IBackupFileSystem`/`BackupFileSystem`** (singleton; a thin `System.IO` wrapper) so the engine's rules and cleanup-on-error are unit-tested against a fake (`FolderPairSynchronizerTests`).
  - **InstantSync (the second profile type) is watcher-driven, not scheduled.** It has its own background driver, **`InstantSyncWatcherService`** (singleton in **three roles** like `BackupSchedulerService` — itself, `IInstantSyncManager`, hosted `BackgroundService`). On startup it attaches a `FileSystemWatcher` to each item of every **enabled** `InstantSync` profile; **`IInstantSyncManager.SyncAsync(profileId)`** (the watcher counterpart of `IBackupScheduler.SyncAsync`) re-reads the profile and (re)attaches or tears down its watchers — **`ProfileService` calls both `scheduler.SyncAsync` and `instantSyncManager.SyncAsync` after every mutation** (each is a no-op for the wrong type), so enabling/disabling/editing/deleting an InstantSync profile starts/stops its watchers immediately. Per item, change events are **debounced** (`DebounceMilliseconds`): events accumulate in pending change/delete sets behind a gate lock, a timer is re-armed on every event, and when it fires after the quiet window the batch is processed on a **single in-flight pass** (a `_processing` flag + a re-arm in the `finally` guarantees no lost events and no overlapping pass — the "handle incoming changes while copying" requirement). Each flush owns **one** profile-associated `OperationLog` (`"Instant Sync '{item}' — {n} change(s)"` rewritten to a summary, exactly like the handler pattern) — but via a **`Logging/DeferredOperationLogger`** so a flush that does no real work writes **no log at all**. (The `{n}` change count is the number of distinct queued path-events, not files copied; a build touching `bin` produces many directory touch-events and short-lived temp files that are no-ops at flush time — directory events whose target already exists, or files that vanished before the debounce settled — so without deferral they'd show up as noisy "synced N change(s) — 0 copied" entries.) The actual copy/delete is **`IInstantSyncProcessor`/`InstantSyncProcessor`** (singleton over `IBackupFileSystem`, unit-tested via `InstantSyncProcessorTests`): each changed source path is rebased under the target and copied **crash-safe** (the same dot-`.tmp`→rename idiom as `FolderPairSynchronizer`; the helper is duplicated, not shared); deleted source paths are mirrored only when `AllowDeletions`. InstantSync is **source-authoritative** (a changed file always overwrites the target — no `OverwriteBehaviour`). The manual **"Run now"** path reuses the runner/handler seam: `InstantSyncHandler` does a one-off full reconcile by mapping each `InstantSyncItem` onto a transient `FolderPair` (`OverwriteBehaviour.AlwaysOverwrite`) and calling the existing `IFolderPairSynchronizer` — so the grid's Run-now works unchanged. **Add a watcher-driven profile type = its own manager/processor/handler in this shape**, registered alongside these.
  - **ArchiveSync (the third profile type) is scheduled** (like FolderPair — it fires from `BackupSchedulerService`, no watcher). Its handler is **`ArchiveSyncHandler`** (`Scheduling/ArchiveSync/`, `Type => ProfileType.ArchiveSync`), same shape as `FolderPairHandler`: one per-run `OperationLog` (`"[Manual] "`-prefixed for Run-now) rewritten in a `finally` to a summary (`"… ran successfully in {d} — {n} archive(s) created, {m} pruned"` / `"… completed with {e} error(s) …"` / `"… failed in {d}"`), per-item try/catch so one item's failure doesn't abort the rest. The actual work is **`IArchiveSyncProcessor`/`ArchiveSyncProcessor`** (singleton over `IBackupFileSystem`, unit-tested via `ArchiveSyncProcessorTests`): per item it (1) builds a ZIP of the source in a **local temp folder** (`IBackupFileSystem.GetTempFilePath` + `CreateZipFromDirectory`, which returns the entry names it added — these are **verbose-logged at `OperationLogLevel.Debug`** as a **single** detail row, `"Archived {n} file(s):"` followed by one file per **CRLF-separated** line, rather than thousands of rows; the file list can't change mid-zip and Debug counts as Info for the header), (2) **crash-safe copies** it into the target as `{FileName}_{yyyy-MM-dd_HHmmss}.zip` (the same dot-`.tmp`→rename idiom, duplicated), then (3) applies **retention**. Timestamps are **local** wall-clock at second precision. **Retention modes** (`ArchiveRetentionMode`): `KeepLastN` deletes everything older than the newest N; **`GrandfatherFatherSon`** keeps N per level across `MaxLevels` levels — each archive's level is encoded in its name (`{FileName}_L{level}_{ts}.zip`) so retention is **derivable from the target folder listing** (a foreign/unparseable file is skipped, never deleted). The GFS algorithm, driven by the per-item **`RunCount`** (incremented only when an archive is actually created), on run index `k`: create the L1 "son"; for `L = 1..MaxLevels-1` (low→high) if `k % N^L == 0` **promote** (rename) the oldest at level `L` up to `L+1` (so a freshly-promoted father can cascade to grandfather the same run); then **trim** each level to N (deleting the aged-out, non-promoted archives; top-level overflow too). This gives true exponential spacing (L1≈last N runs, L2≈N², L3≈N³). The handler advances `RunCount` via a short-lived context after a successful create (mirrors `FolderPairHandler`'s per-pair status write). The manual **"Run now"** path is just `RunAsync(id, manual: true)` through the same runner/handler seam. **Add a scheduled profile type = its own handler (+ processor) in this shape**, registered alongside these.
- `FileSystem/` — `IFolderBrowser`/`FolderBrowser`: lists drives/subfolders on the server for the folder picker (admin-gated; inaccessible dirs skipped). Returns rich entries: `GetDrives()` → `DriveEntry(RootPath, Label)` (Explorer-style label e.g. `Windows (C:)`); `GetDirectories(path)` → `IReadOnlyList<FolderEntry>` (`FullPath`, `Name`, `DateModified`); `GetQuickAccess()` → the Desktop/Documents/Downloads/Pictures shortcuts that exist; `GetParent(path)`; and `CreateDirectory(parentPath, name)` → makes a sub-folder and returns its full path (used by the folder picker's New-folder button). Also `IBackupFileSystem`/`BackupFileSystem` (singleton) — a thin `System.IO`/`System.IO.Compression` wrapper (directory/file existence, list, copy, move, delete, timestamps, content-equal, plus `GetTempFilePath` for a unique local-temp path and `CreateZipFromDirectory` for archiving) used by the folder-pair sync engine (`Scheduling/FolderPairSynchronizer`), the instant-sync copy engine (`Scheduling/InstantSyncProcessor`) and the archive engine (`Scheduling/ArchiveSyncProcessor`) so their logic is testable against a fake; not unit-tested itself. **In-use/locked files are handled, not fatal:** every source read uses a **permissive share mode** (`FileShare.ReadWrite | FileShare.Delete`) so a file another process holds open *for writing* (a log, a VS `.vsidx` index, …) can still be read — plain `File.Copy`/`CreateEntryFromFile` use only `FileShare.Read` and throw "being used by another process" on those. `CopyFile` is therefore a manual stream copy that then **re-stamps the destination's `LastWriteTimeUtc` from the source** (the sync engine compares write-times to decide copy/skip, so a naive copy's "now" timestamp would make every later run see the destination as newer); file *attributes* are intentionally not propagated. `CreateZipFromDirectory` builds the archive entry-by-entry and returns a **`ZipBuildResult`** (`Added` entry names for verbose logging + `Skipped` — `ZipSkippedFile(EntryName, Reason)` — for files that *still* couldn't be read): a single locked file is **skipped, not fatal**, so one bad file never aborts the whole archive. `ArchiveSyncProcessor` logs each skip as a **Warning** detail line and increments the run's error count, so the handler's summary reports "completed with N error(s)" while the archive is still produced from the readable files. A FolderPair/InstantSync copy that still fails on a locked file is caught by the existing crash-safe `CopyThroughTempAsync` (logged Error, temp cleaned, counted) and the rest of the pair continues. **Include/exclude filtering** lives here too: `WildcardMatcher` (static, over the BCL `FileSystemName.MatchesSimpleExpression` — `*`/`?`, case-insensitive, no dependency) and `BackupFilter` (built from a parent's filter rows projected to `FilterRule(FilterDirection, FilterKind, string Pattern)`). **File/Folder matching is name-only** (matched in every sub-folder); **exclude-`Path` matching is by exact relative location**: `IncludesFile` (empty include list ⇒ all files, else the name must match an include), `ExcludesFolder` (a folder-name match skips the subtree), `ExcludesPath(relativeSegments)` (true when the joined relative path equals an exclude-path rule or sits beneath one — that file, or that folder and its subtree; normalised, case-insensitive), `IsFileInScope(name, ancestorFolderNames)` (now also applies `ExcludesPath` to `ancestors + name`) and `IsRelativePathInScope(relPath)` (splits a flat zip path). The folder-pair sync engine threads a `BackupFilter` through its recursion (filtering the copy loop, leaving out-of-scope target files/folders undeleted, and not recursing folders excluded by name *or* by path); the archive engine passes `IsRelativePathInScope` as `CreateZipFromDirectory`'s `includeEntry` predicate.
- `Logging/` — operation logging via a factory: `IOperationLogFactory`/`OperationLogFactory` (singleton, DB-factory) — `CreateAsync(name, level = OperationLogLevel.Info, profileId = null)` inserts a new `OperationLog` row (the `level` here is the **starting/floor** header level — it applies to a detail-less log and is then raised by the detail lines; `profileId` optionally ties the log to a profile so it cascade-deletes with it) and returns an `IOperationLogger` (impl `OperationLogger`) holding that row's id, an internal sequence counter, and the current header level. The logger appends `OperationLogDetail` rows via `AppendAsync(params string[] messages)` (Info) / `AppendAsync(OperationLogLevel level, params string[] messages)` / `ErrorAsync(message, exception?)` (Error) — **one row per message** (pass several in a single call to write several rows rather than one delimited string), all in one short-lived DbContext, auto-incrementing `Sequence` (`Interlocked`); **each detail row carries its own `Level`**, and `ErrorAsync` appends the exception's `Message` only (no stack trace) to the message. **The header `Level` is kept in step as the most severe line seen** — Debug/Info → Info, any Warning → Warning, any Error → Error; the header only rises (it touches the header row only when a line actually escalates it). The header message is **revised in place** via `SetSummaryAsync(message, level)` (updates `OperationLog.Name`, no extra record) — used to turn a "started…" title into a final "succeeded/failed in {duration}" summary so a run is a **single** log event (see `FolderPairHandler`); its `level` is a **floor** that can raise the header (e.g. force Error on a run that failed before logging an error line) but never lowers a level the lines already escalated to. The runtime logger type is `OperationLogger`, deliberately distinct from the `OperationLog` entity (and the enum is `OperationLogLevel`, not `LogLevel`, to avoid the `Microsoft.Extensions.Logging` clash). **`DeferredOperationLogger`** is an `IOperationLogger` wrapper that defers `IOperationLogFactory.CreateAsync` until the first line is actually written, so a unit of work that ends up doing nothing leaves **no** log row (`WasCreated` reports whether anything was written; `SetSummaryAsync` is a no-op until then). Used by the instant-sync watcher to suppress empty per-flush logs. The **read** side is `IOperationLogService`/`OperationLogService` (singleton, DB-factory): `GetPageAsync(page, size, filter?, includeMessages?, level?, profileId?)` → `PagedResult<OperationLog>` headers newest-first (`OrderByDescending(Id)` — SQLite can't `ORDER BY` a `DateTimeOffset`, each with its `Profile` `Include`d), optionally filtered by name (and detail message when `includeMessages`) via `EF.Functions.Like`, by exact `Level`, and by exact `ProfileId` (all AND-combined), and `GetDetailsAsync(logId)` → that log's `OperationLogDetail` lines ordered by `Sequence`. Consumed by the Logs section (`Components/Pages/BackupServicePage/Logs.razor`). **Push notifications:** `ILogWatcher`/`LogWatcher` (singleton) lets views refresh on a change rather than by polling — `OperationLogFactory`/`OperationLogger` call `logWatcher.Notify()` after **every** successful write (header create, detail append, summary rewrite; the watcher is an optional ctor dependency, so test-constructed factories/loggers without it are a no-op). `Notify()` is **coalesced/debounced** (a single timer, `DebounceMilliseconds`): a burst raises **at most one** `Changed` per window, and because the window is **not** re-armed by later writes while one is already pending, a long continuous run still fires `Changed` ~once per window instead of going silent until it stops. `Changed` is raised on a background thread and invokes each subscriber in isolation (a throwing handler can't break the others). The **Logs panel** subscribes in `OnAfterRender(firstRender)` and unsubscribes in `Dispose` (so selecting another sidebar panel — which removes the component — stops the pushes), refreshing the current page **in place** (preserving the user's expanded rows, filters and page) on each `Changed`. **Retention/purge:** `ILogRetentionService`/`LogRetentionService` (singleton; over the DB factory + `TimeProvider` + `ILogger`) owns the `LogRetentionSettings` row (`GetSettingsAsync` lazily seeds the 7/30 default; `UpdateSettingsAsync` clamps each to ≥ 1) and `PurgeIfDueAsync()` deletes log rows older than their retention. `PurgeIfDueAsync` is **guarded to run at most once per (UTC) day** via an in-memory `_lastPurgedUtcDate` + in-flight flag (thread-safe; marks done only on success so a failure retries), and is **fire-and-forget hooked after a log write** — `OperationLogFactory.CreateAsync` and `AuthenticationHistoryService.RecordAsync` both call `_ = logRetentionService?.PurgeIfDueAsync()` (an **optional** ctor dependency, so test-constructed instances skip it). Because **SQLite can't compare a `DateTimeOffset` column**, the purge projects `(Id, TimestampUtc)` into memory, finds the largest `Id` older than the cutoff (`Id` is monotonic with insertion time), and deletes by **`Id` range** (`ExecuteDeleteAsync`) — operation-log details first, then headers. The **Logging** settings section (`Components/Pages/Logging/LoggingPanel`) edits these two values.
- `Profiles/` — `IProfileService`/`ProfileService` (singleton, DB-factory): `CreateAsync(name, description, type, scheduleCron, enabled, IReadOnlyList<FolderPairInput>)` inserts a `Profile` (with its `ProfileType` + `Enabled`) + its folder pairs (`FolderPairInput` is the service-layer DTO carrying `Name`, source/target, `AllowDeletions`, `IncludeSubFolders` and `OverwriteBehaviour`; `Id` 0 = new); `GetPageAsync(...)` reads a sortable page for the grid; `GetAsync(id)` loads one profile (with folder pairs) for editing; `GetSummariesAsync()` returns lightweight `ProfileSummary(Id, Name)` rows (name-ascending) for pickers/filters (e.g. the Logs profile filter); `UpdateAsync(id, …, enabled, IReadOnlyList<FolderPairInput>)` saves profile edits and syncs the folder pairs (updates matched ones by id, adds id-0 ones, removes the rest; the type is fixed and not updated); `SetEnabledAsync(id, enabled)` flips just the `Enabled` flag (used by the grid's inline toggle) and writes a profile-associated operation log named `Profile {name} was enabled/disabled` — a **self-describing** log with the message in the name and **no detail lines** (the convention for simple one-fact events); `DeleteAsync(id)` removes a profile (folder pairs and instant sync items cascade). **`ProfileService` owns only the generic profile fields** (name/description/type/schedule/enabled, its status seeding and the scheduler sync) — the **folder-pair data is split out into `IFolderPairService`/`FolderPairService`** (singleton, no DbContext: a stateless helper over the tracked `Profile.FolderPairs` graph, so a profile save and its folder-pair changes stay one unit of work and one operation log). `ProfileService` injects it and delegates: `Add(profile, inputs)` (build new pairs on create), `Sync(profile, inputs)` (update matched by id / add id-0 / remove the rest, **returning the human-readable change descriptions** the update log appends) and `DescribeForCreateLog(inputs)` (the per-pair detail lines for the create log). Since FolderPair is just one of several profile types, **add a type = its own equivalent data service** alongside `FolderPairService` rather than growing `ProfileService` — `IInstantSyncItemService`/`InstantSyncItemService` (over `Profile.InstantSyncItems`, DTO `InstantSyncInput`) and `IArchiveSyncItemService`/`ArchiveSyncItemService` (over `Profile.ArchiveSyncItems`, DTO `ArchiveSyncInput`; its `Sync` deliberately leaves `RunCount` untouched) are the worked second and third examples. `CreateAsync`/`UpdateAsync` take an `IReadOnlyList<FolderPairInput>` plus optional `IReadOnlyList<InstantSyncInput>` and `IReadOnlyList<ArchiveSyncInput>` lists; `ProfileService` `switch`es on the profile's `Type` and delegates to the matching data service (the other lists stay empty), and `GetAsync` `Include`s all three child collections (with `.ThenInclude(… .Filters)` for FolderPair/ArchiveSync). **Include/exclude filters ride on the FolderPair/ArchiveSync DTOs** as an optional trailing `IReadOnlyList<FilterInput>? Filters` (`FilterInput(Id, FilterDirection, FilterKind, Pattern)`; null = none, so existing call sites are unaffected); `FolderPairService`/`ArchiveSyncItemService` build the child `…Filter` rows on `Add` and reconcile them by id on `Sync` (same matched-update / id-0-add / drop-the-rest pattern as the pairs themselves), and emit filter lines into the create/update operation log. `BackupRunner` likewise `.ThenInclude`s the filters so the engines see them. The profile UI is `Components/Dialogs/ProfileDialog`, used for **both create and edit**: a null `ProfileId` creates, a set one loads + edits. When editing, the `Profile type` dropdown is **hidden** and the type description goes in the title (e.g. "Edit Folder Pairs Profile"); when creating, the dropdown (the `ProfileType` enum) is shown and selects which type-specific editor appears — `FolderPair` renders `FolderPairControl`, `InstantSync` renders `InstantSyncControl` (and disables the Schedule field), `ArchiveSync` renders `ArchiveSyncControl` (Schedule stays enabled — it's scheduled). `ArchiveSyncEditDialog` edits one archive (name, source/target + Browse, archive file name, Include Sub-Folders, a retention-mode `Dropdown` with help text, the keep count, and a Max-levels field shown only for GFS). The dialog also has an `Enabled` checkbox (defaults ticked). Add a profile type = one enum value + a matching control + a `switch` arm in the dialog. Opened from the `Profiles` control, which lists profiles in a sortable, paged grid; an **Enabled** column hosts a `ProfileEnabledControl` (toggle persists inline via `SetEnabledAsync`), and the last column has a `play.png` icon button (**Run now**), a `profile-settings.png` icon button (edit) and a `remove.png` icon button (delete, via `ConfirmDialog`). **Run now** fires `IBackupRunner.RunAsync(id, manual: true)` on a background `Task.Run` (so a long backup doesn't block the circuit; the status-change events refresh the grid as it progresses, and `RunAsync` records its own failures so fire-and-forget is safe) — the run goes through the normal runner path, just logged with a `[Manual]` prefix. The **Status** column reads the live status from `IProfileStatusService` (see below) and is **not** sortable (status isn't a DB column). The run, edit and delete icon buttons are **disabled while that profile is `Running`** (`disabled="@IsRunning(id)"`, re-evaluated on each status-change re-render; `.icon-btn:disabled` greys them) — so a profile can't be run twice from the UI (the runner's `TryBeginRun` is the real guard, but the disabled button avoids the obvious double-click). Opening the edit dialog or the delete-confirmation **locks** the profile via `IProfileStatusService.Lock` (unlocked on cancel/save and in `Dispose`) so a scheduled run won't fire mid-edit — the runner skips a locked profile. `ProfileService` keeps the status tracker in step with the DB: `CreateAsync` seeds the new profile to `Idle`, `DeleteAsync` removes it (which also clears any lock).
  - **`IProfileStatusService`/`ProfileStatusService`** (singleton) — the **in-memory** run-status tracker for profiles (`ProfileStatus` Idle/Running/Error), backed by a `ConcurrentDictionary` (mutated by the background runner, read by the UI). `Get(id)` (defaults to `Idle` for unknown ids), `Set(id, status)` (raises the `event Action<int>? Changed` with the profile id), `Remove(id)`. It also tracks a per-profile **lock** (`Lock`/`Unlock`/`IsLocked`, a separate set) used to mark a profile that's open in an edit/delete dialog — `BackupRunner` skips a locked profile so a scheduled run can't fire mid-edit; `Remove` clears the lock too. The status is also the **single-instance gate**: `IsRunning(id)` and the atomic `TryBeginRun(id)` (sets Running iff not already Running, returns whether it began) enforce **one run per profile at a time** — the scheduler pre-checks `IsRunning` so it won't even spawn a thread, and `BackupRunner` calls `TryBeginRun` as the authority. Status is **never persisted** — on startup `Program.cs` seeds every existing profile to `Idle`. The `Profiles` grid (`@rendermode InteractiveServer`) subscribes to `Changed` in `OnInitialized` and, when the changed profile is on the current page, **reloads the page data** (`InvokeAsync(async () => { await LoadAsync(_page); StateHasChanged(); })`) so the live Status cell *and* the entity-sourced columns it doesn't track (notably `DateLastRun`, which the runner writes in the DB) both refresh as a backup runs (`IDisposable` unsubscribes). The transitions are driven by the scheduler: `BackupRunner` sets `Running` then `Idle`, and `FolderPairHandler`'s catch-all sets `Error` (see `Scheduling/`).
- `Authentication/` — `IAdminCredentialService` / `AdminCredentialService`: seeds the single admin row, verifies login credentials, and changes the password (`ChangePasswordAsync` verifies the current password, then re-hashes), using `PasswordHasher<AdminCredential>` (PBKDF2). Registered as a singleton; uses the DB factory. The settings UI for this lives in `Components/Pages/Authentication/` (`AuthenticationPanel`); its `ChangePasswordDialog` modal lives in `Components/Dialogs/`.
- `IAuthenticationHistoryService` / `AuthenticationHistoryService` (singleton) — writes an `AuthenticationHistory` audit row (`EventType` + UTC timestamp) for each login success/failure (recorded in `Login.razor.cs`) and password change (recorded in `ChangePasswordDialog`), and reads it back paged (`GetPageAsync`, newest first). `AuthenticationPanel` shows this as a paged table below the Change Password button.
- **Auth model:** cookie authentication for a **single admin account** (username `admin`, default password `admin`, stored **hashed** in the `AdminCredentials` table — a future change will allow editing it). Session uses a **30-minute sliding (inactivity) expiry**. Every page requires authentication via `@attribute [Authorize]` in `Components/Pages/_Imports.razor` (applies to all pages there); the login page opts out with `@attribute [AllowAnonymous]`. Enforcement is by `AuthorizeRouteView` in `Routes.razor`, **not** a global authorization `FallbackPolicy` — a fallback policy is enforced at the Blazor endpoint level and would also block `/login`, causing a redirect loop. `Login.razor` is a **static SSR** page (no interactive render mode) so its form POST is a real HTTP request that can write the auth cookie via the cascaded `HttpContext`.
- Configuration comes from `appsettings.json` plus an environment-specific overlay (`appsettings.Development.json` / `appsettings.Production.json`), selected by `ASPNETCORE_ENVIRONMENT` (set to `Development` in `Properties/launchSettings.json`; unset → Production for the deployed service).
- User secrets are enabled (`UserSecretsId` in the csproj) — use `dotnet user-secrets` for local credentials rather than committing them.

## Conventions

- `Nullable` and `ImplicitUsings` are enabled — common namespaces (e.g. `Microsoft.Extensions.Logging`, `System.Threading`) are available without explicit `using` statements.
- The project uses C# primary constructors (see `Worker(ILogger<Worker> logger)`); follow that style for new injected services.
- **Enums live in `BackupService/Enumerations/`** (namespace `BackupService.Enumerations`), one enum per file named after the enum (e.g. `ProfileStatus.cs`). Do not declare enums alongside the entity/component that uses them. The namespace is brought in via `@using BackupService.Enumerations` in `Components/_Imports.razor` for the component tree, and an explicit `using BackupService.Enumerations;` in non-Razor files that reference an enum.
- **Extension methods live in `BackupService/Extensions/`** (namespace `BackupService.Extensions`), e.g. `EnumExtensions.GetDescription(this Enum)` which returns a value's `[Description(...)]` text (falling back to the name). Give enum values shown in the UI a `[Description]` and render `value.GetDescription()` rather than the raw name. There is also a project-local `[HelpText("…")]` attribute (`Extensions/HelpTextAttribute.cs`) read by `EnumExtensions.GetHelpText(this Enum)` (returns the text, or empty string if absent) — used for the longer explanatory blurb shown under a dropdown (e.g. `OverwriteBehaviour`). The namespace is imported in `Components/_Imports.razor`. (Note: in the test project NUnit's global `using NUnit.Framework` makes `Description` ambiguous with `System.ComponentModel.DescriptionAttribute` — fully-qualify it there.)
- **Blazor components use code-behind.** A `.razor` file should hold only markup and the minimal C# unavoidable in markup (e.g. simple `@bind`/event wiring); put the bulk of the logic — fields, properties, lifecycle methods, event handlers, `[Inject]` services — in a sibling `.razor.cs` partial class (`public partial class Foo { ... }` in the component's namespace). Create the code-behind whenever a component needs more than trivial inline code.
- **Each page gets its own folder under `Components/Pages/`.** A routable page lives in `Components/Pages/<PageName>/<PageName>.razor`, with its code-behind and any page-specific files (sub-components, view models, partials) in that same folder. Examples: `Components/Pages/Dashboard/`, `Components/Pages/Settings/`, `Components/Pages/Authentication/` (the Login page). The folder becomes part of the component namespace (e.g. `BackupService.Components.Pages.Authentication`); match the code-behind's `namespace` to it. Routes come from `@page`, not the folder, so moving a page between folders doesn't change its URL. (Note: the page folder `Components/Pages/Authentication/` is distinct from the root `Authentication/` folder, which holds the auth *services*.)
- **For logging, use `Microsoft.Extensions.Logging`** — inject `ILogger<T>` via the constructor (already configured by the host; no third-party logging framework like Serilog or NLog). Use structured-logging message templates (e.g. `logger.LogInformation("Backed up {File}", path)`).
- **Use dependency injection throughout, via the built-in `Microsoft.Extensions.DependencyInjection` container** (available as `builder.Services` — no third-party container like Autofac). Register services on `builder.Services` (`AddSingleton`/`AddScoped`/`AddTransient`) and inject them via constructors rather than newing them up or using static/singleton access. Backup logic should live in injectable services (interface + implementation) consumed by `Worker`, not inline in `Worker` itself.
