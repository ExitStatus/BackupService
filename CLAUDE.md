# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is a **.NET 10 ASP.NET Core app (`Microsoft.NET.Sdk.Web`) hosting Blazor Server plus a background worker**, designed to run as a **Windows Service** (`builder.Host.UseWindowsService()`). It started life as a Worker Service template and was converted: the same process now serves an interactive Blazor Server UI *and* runs `Worker` as a hosted background service. The actual backup functionality has not been implemented yet — `Worker.cs` still just logs a heartbeat — so most work here is greenfield.

## Commands

Run from the repo root (`C:\Projects\BackupService`). The solution is `BackupService.slnx` (the newer XML solution format); the project lives in `BackupService/`.

```powershell
dotnet build                          # build the solution
dotnet run --project BackupService    # run web host + worker (Ctrl+C to stop)
dotnet watch --project BackupService run   # run with hot reload
```

Running interactively (console / `dotnet run`) starts Kestrel and serves the Blazor UI. `UseWindowsService()` only changes the host lifetime when the process is actually started by the Windows SCM, so local console runs are unaffected. To install as a Windows Service, publish then run `BackupService.exe --install` from an elevated prompt (see the install commands below).

**HTTP only — deployed on `http://localhost:55000`, dev on `http://localhost:5080`.** There is no HTTPS (the admin UI is local-only). The port lives in **environment-specific** config (not base `appsettings.json`): `appsettings.Production.json` → `Kestrel:Endpoints:Http:Url` = 55000 (the deployed service runs in Production), `appsettings.Development.json` → 5080. Base `appsettings.json` intentionally has **no** `Kestrel:Endpoints` — otherwise Visual Studio reads it as the default and launches the browser on 55000 even in debug. The dev launch profile (`launchSettings.json`) sets `ASPNETCORE_ENVIRONMENT=Development` (web-standard) and `applicationUrl=5080`. Note a `Kestrel:Endpoints` value takes precedence over `ASPNETCORE_URLS`/`applicationUrl`, so the per-environment port is set via the config layer.

**Install as a Windows Service:** run the published exe elevated — `BackupService.exe --install` (registers autostart + LocalSystem, replaces an existing service, and starts it) or `BackupService.exe --uninstall` (stops + removes it). These commands run and exit without starting the web host; see `Hosting/WindowsServiceInstaller.cs` (shells out to `sc.exe`). Must be run from the published exe, not `dotnet …dll`.

**Deploy via publish profile:** `Properties/PublishProfiles/Install.pubxml` publishes to `C:\Tools\BackupService` and (re)installs the service — `dotnet publish -p:PublishProfile=Install`. It stops the running service before copying (so files aren't locked) and runs `--install` afterward. **Run it from an elevated prompt**; otherwise the deploy still succeeds but the install step only warns (it needs admin). The ASP.NET Core dev HTTPS certificate / "trust the self-signed cert" prompt is irrelevant here and does not apply to the service. If the UI ever needs network/HTTPS access, configure a real cert under `Kestrel:Endpoints` and add `CookieSecurePolicy.Always` + `UseHttpsRedirection()`.

Note when running the built dll directly for testing: set the working directory to the output folder, otherwise the content root is the current directory and `appsettings.json` (hence the Kestrel binding) won't be found. The real service is unaffected because `UseWindowsService()` sets the content root to the exe's directory.

Tests live in the sibling **`BackupService.UnitTests`** project (in the solution). Stack: **NUnit** + **Moq** + **FluentAssertions**. Note FluentAssertions is pinned to **7.x** (the last Apache-2.0/free line — 8.x requires a paid commercial license); don't bump to 8.x without a license.

```powershell
dotnet test                                                   # run all tests
dotnet test --filter "FullyQualifiedName~TestClassName.TestMethodName"   # single test
```

Testing services that use the DbContext factory: mock `IDatabaseContextFactory` with Moq and have `CreateDbContext()` return a `BackupDbContext` over a **shared in-memory SQLite** connection (`new SqliteConnection("DataSource=:memory:")` kept open for the test, `EnsureCreated()` for schema) — see `Authentication/AdminCredentialServiceTests.cs` for the pattern. Use `NullLogger<T>.Instance` for `ILogger` dependencies.

### Database / EF Core migrations

Migrations are kept under `BackupService/Database/Migrations`. Always pass `--output-dir Database/Migrations` when adding one so it lands there rather than the default `Migrations/` folder. The `dotnet-ef` tool version must match the EF Core packages (currently 10.0.9).

```powershell
dotnet ef migrations add <Name> --project BackupService --output-dir Database/Migrations
dotnet ef database update --project BackupService     # apply migrations manually
dotnet ef migrations remove --project BackupService   # undo the last (unapplied) migration
```

The app also calls `db.Database.Migrate()` on startup, so pending migrations apply automatically when it runs. The SQLite file (`backupservice.db`, plus `-wal`/`-shm`) is generated at the content root and is not source.

## Architecture

- `Program.cs` — host bootstrap via `WebApplication.CreateBuilder`. Calls `UseWindowsService()`, registers the DB factory, cookie auth + authorization, Razor Components with interactive server rendering (`AddRazorComponents().AddInteractiveServerComponents()`), registers the backup scheduler as the hosted service, then maps the Blazor endpoint (`MapRazorComponents<App>().AddInteractiveServerRenderMode()`) and a `/logout` POST endpoint. The middleware pipeline is `UseAuthentication` → `UseAuthorization` → `UseAntiforgery` → `MapStaticAssets` → endpoints. Startup also runs `db.Database.Migrate()` and `IAdminCredentialService.EnsureSeededAsync()`. **Static assets use .NET fingerprinting:** `app.MapStaticAssets()` (not `UseStaticFiles`) serves `wwwroot`, and `App.razor` links the stylesheet via `@Assets["app.css"]` so the URL carries a content hash — CSS/JS edits bust the browser cache automatically (avoids the stale-CSS problem where new markup loads but old styles persist).
- **Backup execution is driven by the scheduler, not a `Worker`.** (`Worker.cs` — the old idle heartbeat placeholder — has been removed; `BackupSchedulerService` is the `BackgroundService` now.) Backup logic belongs in the per-type handlers in `Scheduling/` (see below), not inline in the host.
- `Database/` — **all** database code lives here: `BackupDbContext` (EF Core, SQLite), the entity models, the `Migrations/` subfolder, `BackupDatabaseLocation`, and the `DatabaseContextFactory`. Keep new entities and DbContext changes in this folder. Core domain: a `Profile` has one-to-many `FolderPair` records (cascade delete); the `FolderPair` carries status enums (`FolderPairStatus`, `FolderPairLastRunStatus`). **A profile's own run status (`ProfileStatus` — Idle/Running/Error) is deliberately NOT persisted** — it describes the current run and is tracked in memory by `IProfileStatusService` (see `Profiles/`); there is no `Profile.Status` column. Each `FolderPair` also has an `OverwriteBehaviour` enum (how to treat an existing/newer file at the destination). `Profile` has a `bool Enabled` (on/off switch) that **defaults to true** via a store default — `BackupDbContext.OnModelCreating` configures `Property(p => p.Enabled).HasDefaultValue(true)` so the migration backfills existing rows to enabled; the C# initializer is also `= true`. (EF Core 7+ then treats `true` as the sentinel, so a `false` value is still persisted — disabling a profile works.) Other tables: `AdminCredential`, `AuthenticationHistory`, `BackupRecord`. There is also an operation-logging pair: `OperationLog` (header — unbounded `Name`, `TimestampUtc`, and an `OperationLogLevel` enum — Info/Warning/Error/Debug — the **severity of the operation as a whole**, set when the log is created) has one-to-many `OperationLogDetail` rows (cascade delete; unbounded `Message`, `TimestampUtc`, and an `int Sequence` for ordering lines within a log — detail rows carry **no** level of their own). `OperationLog` also has a **nullable** `int? ProfileId` FK to `Profile` (null for logs not tied to a profile) configured in `OnModelCreating` with `OnDelete(DeleteBehavior.Cascade)` — deleting a profile cascade-deletes its logs (cascade is the non-default behaviour for a nullable FK, so it's set explicitly). Note the profile *deletion* log is deliberately left unassociated (`ProfileId` null) so it survives the delete. Written via the `Logging/` factory; read/displayed by the Logs sidebar section.
  - **Factory pattern — the `DbContext` is NOT registered in DI.** Instead `IDatabaseContextFactory` (impl `DatabaseContextFactory`) is registered as a singleton. Code that needs the database injects `IDatabaseContextFactory` and calls `CreateDbContext()`, owning and disposing a short-lived context per unit of work (`using var db = factory.CreateDbContext();`). Do **not** add `AddDbContext<BackupDbContext>` back. `DatabaseContextFactory` also implements `IDesignTimeDbContextFactory<BackupDbContext>` so `dotnet ef` can build a context at design time without the context being in DI.
  - **SQLite caveat:** SQLite cannot `ORDER BY` (or compare) a `DateTimeOffset` column. For "newest first" ordering use the autoincrement `Id` (monotonic with insertion) rather than a timestamp column — see `AuthenticationHistoryService.GetPageAsync`. `DateTimeOffset` columns are fine to store and `SELECT`/display.
  - **Single database, location depends on hosting** (`BackupDatabaseLocation.GetConnectionString()`): there is only ever one SQLite file. When running as a Windows Service it lives under ProgramData (`%ProgramData%\BackupService\backupservice.db`) so it's shared machine-wide regardless of the service account; when run interactively (debugging/console) it sits next to the exe (`AppContext.BaseDirectory`). The decision uses `WindowsServiceHelpers.IsWindowsService()`. There is no connection string in `appsettings.json` — it is computed in code.
- `Components/` — the Blazor Server UI. `App.razor` is the root HTML document (loads `_framework/blazor.web.js` and the theme `app.css`), `Routes.razor` wires the router via `AuthorizeRouteView` (unauthenticated → `RedirectToLogin`), and routable pages live in `Components/Pages/` (`@page` directives). `_Imports.razor` holds shared `@using`s for the component tree. Two layouts: `Components/Layout/MainLayout.razor` (default — top bar with signed-in user + Log out, gated by `<AuthorizeView>`, content in `.page`) and `Components/Layout/AuthLayout.razor` (full-screen centered, used by `Login` via `@layout AuthLayout`).
- **Layout shell:** `MainLayout` wraps the top bar + `<main class="app-main">` in `.app-shell` (a flex column at `height: 100vh`), so `.app-main` fills the viewport below the title bar. Pages that want a centred, max-width look opt into a `.page` wrapper; pages that need full height/bleed use `.app-main` directly.
- **Workspace pages (sidebar + content):** both the home **`BackupServicePage`** (route `/`, title "Backup Service") and the **Settings** page use the shared `.workspace` / `.sidebar` / `.sidebar-item` / `.workspace-content` CSS: a left-docked navigation sidebar (a self-contained control owning its items, two-way `@bind-SelectedKey`) plus a content area that swaps the selected section's control. `BackupServicePage` has `BackupServiceSidebar` (items: Dashboard → `Dashboard.razor`, Backup Profiles → `Profiles.razor`, Logs → `Logs.razor`); Settings has `SettingsSidepanel` (item: Authentication → `AuthenticationPanel`). Add a section = one sidebar item + a matching control. The **Logs** section is a paged table of `OperationLog` headers (newest first, via `IOperationLogService`) — each row shows the header's level as a coloured `.log-level` pill, the associated **Profile** name (or a muted `—` when not tied to one), the log name, and timestamp, plus a `+`/`−` button that expands a **terminal-style view** of that log's `OperationLogDetail` lines — monospace `[{timestamp}] {message}` rows (no wrap, like a real terminal; long messages scroll horizontally), lazy-loaded on first expand and cached. Because a big backup can write thousands of lines, the lines are rendered with `<Virtualize>` (only visible rows are in the DOM) inside a `.log-terminal` box capped at ~20 lines (`max-height` = 20 × the 20px `.log-terminal-line` height, matched by `LineHeight`/`ItemSize` in `Logs.razor.cs`) that scrolls vertically beyond that. The detail cache is a `List<OperationLogDetail>` so it can feed `Virtualize.Items`. The expand control only shows when the log actually has detail lines — `GetPageAsync` populates a non-persisted `OperationLog.DetailCount` ([NotMapped], via a grouped count query for the page) and the grid hides the `+` when it's 0, so self-describing logs (message-in-name, no sub-entries) render as a plain row. Above the table a `.filter-bar` has a free-text name filter (live, `@oninput`), an **Include Messages** checkbox, a level `Dropdown` (`OperationLogLevel?`, null = "All levels"), and a profile `Dropdown` (`int?` profile id, null = "All profiles"; options come from `IProfileService.GetSummariesAsync`). `GetPageAsync(page, size, filter, includeMessages, level, profileId)` does the matching server-side — name (and detail `Message` too when the box is ticked) via `EF.Functions.Like` (LIKE is case-insensitive for ASCII in SQLite), an exact `Level` match, and an exact `ProfileId` match; all filters combine with AND, and each returned log has its `Profile` navigation `Include`d for the column.
- **Naming caveat:** never name a type or page folder `BackupService` — it equals the root namespace and shadows it (breaks `@using BackupService...`). The home page folder/class is `BackupServicePage` for this reason.
- **Interactivity:** pages are static SSR by default. Components needing interactivity opt in with `@rendermode InteractiveServer` (infrastructure already wired in `Program.cs`). The Settings page is the first such page (the `SettingsSidepanel` nav updates the shown section via `@bind-SelectedKey`). Keep the login page static SSR (it must write the auth cookie on its form POST).
- **Theme/styling:** a single global dark theme in `wwwroot/app.css`, driven by CSS custom properties under `:root` (`--bg`, `--surface`, `--accent`, …) — re-skin by editing those variables. Shared classes: `.topbar`, `.page`, `.card`, `.auth-shell`/`.auth-card`, `.btn-secondary`, `.alert-error`. No CSS framework (no Bootstrap). Note: `wwwroot` is served via `app.MapStaticAssets()` (the Web SDK static-web-assets system with fingerprinting) — live from source in Development and copied into `wwwroot` on `publish`. Reference the stylesheet via `@Assets["app.css"]` (already done in `App.razor`) so the served URL is content-hashed; a plain `href="app.css"` caches indefinitely and shows stale styles after edits. (A raw `dotnet bin\...dll` run in Production still can't serve `/app.css` because the assets aren't copied to `bin/` on a plain `build` — a test artifact only, not a real-deployment issue.)
- `Components/Controls/` — shared, reusable custom controls (namespace `BackupService.Components.Controls`, imported globally via `Components/_Imports.razor`). **Modal dialogs are not here — they live in `Components/Dialogs/` (see below).** `Notification` — a non-modal toast: call `Show(message, NotificationLevel)` via `@ref`; shows bottom-right for 8s then fades (Success/Warning/Error → green/amber/red). `Dropdown<TValue>` — theme-controlled dropdown that **replaces the native `<select>`** (whose OS-drawn popup ignores the dark theme on Windows — do not use raw `<select>`; use this). Supports `@bind-Value`, takes `Items` + a `DisplayText` func, optional `Disabled`; closes on option/backdrop click. `ToggleButton` — a sliding on/off switch (knob left = `false`, right = `true`); two-way bindable via `@bind-Value` (bool), optional `Disabled`. `ProfileEnabledControl` — composes a `ToggleButton` with a status swatch (green when true, red when false); `@bind-Value` + raises `ValueChanged` when toggled (used in the Profiles grid's Enabled column to persist inline via `IProfileService.SetEnabledAsync`). `TimePicker` — themed **clock-face time picker** (Material-style) replacing the native `<input type="time">` (whose OS popup ignores the dark theme): a trigger shows the current time and opens a popup with a 12-hour clock you click to set the hour (which auto-advances to the minute), an AM/PM toggle, and a clickable HH:MM header to switch between setting hour/minute; two-way bindable via `@bind-Value` (a `TimeOnly`), applies on OK. Used by `ScheduleDialog` for the Daily/Weekly/Monthly "At time" field. Clock geometry constants in the code-behind must match the `.tp-clock` size in `app.css`; clicks are read from `MouseEventArgs.OffsetX/Y` so all overlay children (numbers/hands/knob) are `pointer-events: none`. `FolderPairControl` — manages a profile's **list** of folder pairs: a table of pairs (name/watched — the source/target paths can be long, so the list shows the name) with per-row edit/delete and an Add button, bound to a `List<FolderPairModel>` it mutates; exposes a `Validate()` (requires ≥1 pair) the parent calls before submit. It opens `FolderPairEditDialog` (in `Components/Dialogs/`) to add/edit a pair. The `FolderPairModel` view model lives here alongside `FolderPairControl`.
- `Components/Dialogs/` — **all modal dialog components** (namespace `BackupService.Components.Dialogs`, imported globally via `Components/_Imports.razor`). New dialogs go here, not in `Controls/` or a page folder. `FolderBrowserDialog` — modal server-side folder picker styled like the **Windows Explorer "Select Folder" dialog** (themed dark): window title bar, toolbar with back/forward/up + an address-bar breadcrumb (clickable path segments), a left navigation tree (Quick access + This PC → drives, lazy-expand), a details list with Name/Date modified/Type/Size columns (single-click selects + fills the Folder box, double-click drills in), and a bottom Folder textbox + Select Folder/Cancel. Uses `IFolderBrowser`; takes `InitialPath`, raises `OnSelect(string)`/`OnCancel`. The tree rows are a recursive `FolderTreeNode` control (lazy-loads children on expand). `ScheduleDialog` — modal schedule builder producing a `ScheduleDefinition`. `ConfirmDialog` — reusable modal confirmation (`Title`/`Message`/`ConfirmText`, `OnConfirm`/`OnCancel`); confirm button uses the `.btn-danger` style. `FolderPairEditDialog` — modal that adds/edits a single pair (name + source/target with Browse + watch flag + an **Overwrite behaviour** `Dropdown` whose selected value's `GetHelpText()` is shown underneath as `.help-text`), hosts the folder browser, and returns the edited `FolderPairModel` (the model + the list-level `FolderPairControl` stay in `Components/Controls/`). `ProfileDialog` — create/edit a backup profile (hosts `ScheduleDialog` and the type-specific editor, e.g. `FolderPairControl`); see the `Profiles/` entry for its behaviour. `ChangePasswordDialog` — self-contained admin password-change modal (uses `IAdminCredentialService` + `IAuthenticationHistoryService`). The schedule is a profile-level field collected by `ProfileDialog` (which hosts the `ScheduleDialog`), not by `FolderPairEditDialog`.
- `Scheduling/` — `ScheduleMode` + `ScheduleDefinition` (the user-friendly schedule selection). `ToCron()` makes a standard 5-field cron string (stored in `Profile.Schedule`), `ToHumanReadable()` makes the label. **The raw cron is never shown to the user — always humanize.** `FromCron(cron)` parses the stored cron back into a definition (used to pre-fill the edit dialog's schedule builder) and `Describe(cron)` is the always-safe display helper ("Not scheduled" / friendly label / raw cron only as a last-resort fallback) used by the grid. Cron is validated/parsed with the **Cronos** package (also used for next-run times by the scheduler below).
  - **The scheduler that actually fires profiles** also lives here. `BackupSchedulerService` is a singleton registered in **three roles** — itself, `IBackupScheduler`, and the hosted `BackgroundService` (one shared instance via `AddSingleton(...)` + `AddSingleton<IBackupScheduler>(sp => sp.GetRequiredService<...>())` + `AddHostedService(sp => ...)`). On startup it loads every enabled profile that has a `Schedule` and registers it; a loop computes each profile's next run with Cronos (`CronExpression.GetNextOccurrence(DateTimeOffset.Now, TimeZoneInfo.Local)` — **local wall-clock, DST-correct**), waits until the soonest (capped at 1 min, and woken early on change via a `SemaphoreSlim`), then fires every due profile. **Jobs run concurrently** (each due profile launched on its own `Task.Run`, so a slow backup never blocks others) and a profile is **never run twice at once** — before spawning a thread the scheduler skips a profile whose run is still in progress (`IProfileStatusService.IsRunning`), and `BackupRunner` re-checks atomically via `TryBeginRun` (the authority). It reads profiles via `IDatabaseContextFactory` directly (not `IProfileService`) to avoid a DI cycle. The `IBackupScheduler` API is a single `SyncAsync(profileId)` that re-reads the profile and schedules it (enabled + parseable cron) or unschedules it (missing/disabled/blank/invalid) — `ProfileService` calls it after **every** mutation (`CreateAsync`/`UpdateAsync`/`SetEnabledAsync`/`DeleteAsync`) so edits take effect immediately. (`GetNextOccurrence(cron, from, tz)` is a pure static helper, extracted for tests.)
  - **What runs when a schedule fires is dispatched by `ProfileType`.** `IBackupRunner`/`BackupRunner` (singleton) is the dispatcher: `RunAsync(profileId)` loads the profile **with its `FolderPairs`**, **skips the run if the profile is locked** (`IProfileStatusService.IsLocked` — an admin has it open in an edit/delete dialog), otherwise sets the in-memory status to `Running`, runs the handler, **stamps `DateLastRun` (the only DB write — status is no longer persisted) and *then* flips the status to `Idle`** (order matters: the grid reloads on the status change, so the timestamp must be saved first or the grid shows the previous run's `DateLastRun`). On failure the handler has already set `Error`; the runner stamps `DateLastRun` and re-asserts `Error` so the grid's status-triggered reload also picks up the fresh timestamp. It does **not** write its own operation log — **the handler owns the single per-run log** (see next). It calls the `IProfileTypeHandler` whose `Type` matches `profile.Type` (handlers injected as `IEnumerable<IProfileTypeHandler>`, indexed into a dictionary); an unmapped type is the one case the runner itself logs (a "Scheduled backup failed" error log) since no handler runs. On a handler exception the runner just records the failure via `ILogger` (the handler already set `Error` status and logged — see next). `IProfileTypeHandler` is the per-type seam (`ProfileType Type { get; }` + `HandleAsync(Profile, ct)`); `FolderPairHandler` (`Type => ProfileType.FolderPair`) is the only one so far and **currently just logs that it ran** (the real source→target copy logic lands here next). It creates **exactly one** `OperationLog` per run: the header starts as `"{Type.GetDescription()} Handler called with {n} folder pair(s)."` (e.g. "Folder Pairs Handler called with 2 folder pair(s)."), the folder pairs are appended as detail lines, and a `finally` rewrites the header (via `SetSummaryAsync`) to `"… Handler ran successfully in {duration}"` (Info) or `"… Handler failed in {duration}"` (Error). The "{ProfileType description} Handler" prefix is derived from the enum's `[Description]` (not a hardcoded string), and the log already carries the profile via its FK, so the message doesn't repeat the profile id/name. A failure also sets the profile status to `Error` (via `IProfileStatusService`) and re-throws. **Add a profile type = one `IProfileTypeHandler` impl + its `AddSingleton<IProfileTypeHandler, …>()` registration** — no scheduler/runner change (mirrors the `ProfileDialog` "add a type" convention); each handler is expected to own its single per-run log.
- `FileSystem/` — `IFolderBrowser`/`FolderBrowser`: lists drives/subfolders on the server for the folder picker (admin-gated; inaccessible dirs skipped). Returns rich entries: `GetDrives()` → `DriveEntry(RootPath, Label)` (Explorer-style label e.g. `Windows (C:)`); `GetDirectories(path)` → `IReadOnlyList<FolderEntry>` (`FullPath`, `Name`, `DateModified`); `GetQuickAccess()` → the Desktop/Documents/Downloads/Pictures shortcuts that exist; plus `GetParent(path)`.
- `Logging/` — operation logging via a factory: `IOperationLogFactory`/`OperationLogFactory` (singleton, DB-factory) — `CreateAsync(name, level = OperationLogLevel.Info, profileId = null)` inserts a new `OperationLog` row (the **level is set here, on the header** — it is the severity of the operation as a whole, not per line; `profileId` optionally ties the log to a profile so it cascade-deletes with it) and returns an `IOperationLogger` (impl `OperationLogger`) holding that row's id and an internal sequence counter. The logger appends `OperationLogDetail` rows via `AppendAsync(params string[] messages)` / `ErrorAsync(message, exception?)` — **one row per message** (pass several in a single call to write several rows rather than one delimited string), all in one short-lived DbContext, auto-incrementing `Sequence` (`Interlocked`); detail rows carry only the message (no level), and `ErrorAsync` appends the exception's full `ToString()` (incl. stack trace) to the message. The header's message/level start at creation but can be **revised in place** via `SetSummaryAsync(message, level)` (updates `OperationLog.Name` + `Level`, no extra record) — used to turn a "started…" title into a final "succeeded/failed in {duration}" summary at the matching level so a run is a **single** log event (see `FolderPairHandler`). The runtime logger type is `OperationLogger`, deliberately distinct from the `OperationLog` entity (and the enum is `OperationLogLevel`, not `LogLevel`, to avoid the `Microsoft.Extensions.Logging` clash). The **read** side is `IOperationLogService`/`OperationLogService` (singleton, DB-factory): `GetPageAsync(page, size, filter?, includeMessages?, level?, profileId?)` → `PagedResult<OperationLog>` headers newest-first (`OrderByDescending(Id)` — SQLite can't `ORDER BY` a `DateTimeOffset`, each with its `Profile` `Include`d), optionally filtered by name (and detail message when `includeMessages`) via `EF.Functions.Like`, by exact `Level`, and by exact `ProfileId` (all AND-combined), and `GetDetailsAsync(logId)` → that log's `OperationLogDetail` lines ordered by `Sequence`. Consumed by the Logs section (`Components/Pages/BackupServicePage/Logs.razor`).
- `Profiles/` — `IProfileService`/`ProfileService` (singleton, DB-factory): `CreateAsync(name, description, type, scheduleCron, enabled, IReadOnlyList<FolderPairInput>)` inserts a `Profile` (with its `ProfileType` + `Enabled`) + its folder pairs (`FolderPairInput` is the service-layer DTO carrying `Name`, source/target, `WatchFolder`, `AllowDeletions` and `OverwriteBehaviour`; `Id` 0 = new); `GetPageAsync(...)` reads a sortable page for the grid; `GetAsync(id)` loads one profile (with folder pairs) for editing; `GetSummariesAsync()` returns lightweight `ProfileSummary(Id, Name)` rows (name-ascending) for pickers/filters (e.g. the Logs profile filter); `UpdateAsync(id, …, enabled, IReadOnlyList<FolderPairInput>)` saves profile edits and syncs the folder pairs (updates matched ones by id, adds id-0 ones, removes the rest; the type is fixed and not updated); `SetEnabledAsync(id, enabled)` flips just the `Enabled` flag (used by the grid's inline toggle) and writes a profile-associated operation log named `Profile {name} was enabled/disabled` — a **self-describing** log with the message in the name and **no detail lines** (the convention for simple one-fact events); `DeleteAsync(id)` removes a profile (folder pairs cascade). The profile UI is `Components/Dialogs/ProfileDialog`, used for **both create and edit**: a null `ProfileId` creates, a set one loads + edits. When editing, the `Profile type` dropdown is **hidden** and the type description goes in the title (e.g. "Edit Folder Pairs Profile"); when creating, the dropdown (the `ProfileType` enum) is shown and selects which type-specific editor appears — `FolderPair` renders `FolderPairControl`. The dialog also has an `Enabled` checkbox (defaults ticked). Add a profile type = one enum value + a matching control + a `switch` arm in the dialog. Opened from the `Profiles` control, which lists profiles in a sortable, paged grid; an **Enabled** column hosts a `ProfileEnabledControl` (toggle persists inline via `SetEnabledAsync`), and the last column has a `profile-settings.png` icon button (edit) and a `remove.png` icon button (delete, via `ConfirmDialog`). The **Status** column reads the live status from `IProfileStatusService` (see below) and is **not** sortable (status isn't a DB column). The edit and delete icon buttons are **disabled while that profile is `Running`** (`disabled="@IsRunning(id)"`, re-evaluated on each status-change re-render; `.icon-btn:disabled` greys them). Opening the edit dialog or the delete-confirmation **locks** the profile via `IProfileStatusService.Lock` (unlocked on cancel/save and in `Dispose`) so a scheduled run won't fire mid-edit — the runner skips a locked profile. `ProfileService` keeps the status tracker in step with the DB: `CreateAsync` seeds the new profile to `Idle`, `DeleteAsync` removes it (which also clears any lock).
  - **`IProfileStatusService`/`ProfileStatusService`** (singleton) — the **in-memory** run-status tracker for profiles (`ProfileStatus` Idle/Running/Error), backed by a `ConcurrentDictionary` (mutated by the background runner, read by the UI). `Get(id)` (defaults to `Idle` for unknown ids), `Set(id, status)` (raises the `event Action<int>? Changed` with the profile id), `Remove(id)`. It also tracks a per-profile **lock** (`Lock`/`Unlock`/`IsLocked`, a separate set) used to mark a profile that's open in an edit/delete dialog — `BackupRunner` skips a locked profile so a scheduled run can't fire mid-edit; `Remove` clears the lock too. The status is also the **single-instance gate**: `IsRunning(id)` and the atomic `TryBeginRun(id)` (sets Running iff not already Running, returns whether it began) enforce **one run per profile at a time** — the scheduler pre-checks `IsRunning` so it won't even spawn a thread, and `BackupRunner` calls `TryBeginRun` as the authority. Status is **never persisted** — on startup `Program.cs` seeds every existing profile to `Idle`. The `Profiles` grid (`@rendermode InteractiveServer`) subscribes to `Changed` in `OnInitialized` and, when the changed profile is on the current page, **reloads the page data** (`InvokeAsync(async () => { await LoadAsync(_page); StateHasChanged(); })`) so the live Status cell *and* the entity-sourced columns it doesn't track (notably `DateLastRun`, which the runner writes in the DB) both refresh as a backup runs (`IDisposable` unsubscribes). The transitions are driven by the scheduler: `BackupRunner` sets `Running` then `Idle`, and `FolderPairHandler`'s catch-all sets `Error` (see `Scheduling/`).
- `Authentication/` — `IAdminCredentialService` / `AdminCredentialService`: seeds the single admin row, verifies login credentials, and changes the password (`ChangePasswordAsync` verifies the current password, then re-hashes), using `PasswordHasher<AdminCredential>` (PBKDF2). Registered as a singleton; uses the DB factory. The settings UI for this lives in `Components/Pages/Authentication/` (`AuthenticationPanel`); its `ChangePasswordDialog` modal lives in `Components/Dialogs/`.
- `IAuthenticationHistoryService` / `AuthenticationHistoryService` (singleton) — writes an `AuthenticationHistory` audit row (`EventType` + UTC timestamp) for each login success/failure (recorded in `Login.razor.cs`) and password change (recorded in `ChangePasswordDialog`), and reads it back paged (`GetPageAsync`, newest first). `AuthenticationPanel` shows this as a paged table below the Change Password button.
- **Auth model:** cookie authentication for a **single admin account** (username `admin`, default password `admin`, stored **hashed** in the `AdminCredentials` table — a future change will allow editing it). Session uses a **30-minute sliding (inactivity) expiry**. Every page requires authentication via `@attribute [Authorize]` in `Components/Pages/_Imports.razor` (applies to all pages there); the login page opts out with `@attribute [AllowAnonymous]`. Enforcement is by `AuthorizeRouteView` in `Routes.razor`, **not** a global authorization `FallbackPolicy` — a fallback policy is enforced at the Blazor endpoint level and would also block `/login`, causing a redirect loop. `Login.razor` is a **static SSR** page (no interactive render mode) so its form POST is a real HTTP request that can write the auth cookie via the cascaded `HttpContext`.
- Configuration comes from `appsettings.json` plus an environment-specific overlay (`appsettings.Development.json` / `appsettings.Production.json`), selected by `ASPNETCORE_ENVIRONMENT` (set to `Development` in `Properties/launchSettings.json`; unset → Production for the deployed service).
- User secrets are enabled (`UserSecretsId` in the csproj) — use `dotnet user-secrets` for local credentials rather than committing them.

## Conventions

- `Nullable` and `ImplicitUsings` are enabled — common namespaces (e.g. `Microsoft.Extensions.Logging`, `System.Threading`) are available without explicit `using` statements.
- The project uses C# primary constructors (see `Worker(ILogger<Worker> logger)`); follow that style for new injected services.
- **Enums live in `BackupService/Enumerations/`** (namespace `BackupService.Enumerations`), one enum per file named after the enum (e.g. `ProfileStatus.cs`). Do not declare enums alongside the entity/component that uses them. The namespace is brought in via `@using BackupService.Enumerations` in `Components/_Imports.razor` for the component tree, and an explicit `using BackupService.Enumerations;` in non-Razor files that reference an enum.
- **Extension methods live in `BackupService/Extensions/`** (namespace `BackupService.Extensions`), e.g. `EnumExtensions.GetDescription(this Enum)` which returns a value's `[Description(...)]` text (falling back to the name). Give enum values shown in the UI a `[Description]` and render `value.GetDescription()` rather than the raw name. There is also a project-local `[HelpText("…")]` attribute (`Extensions/HelpTextAttribute.cs`) read by `EnumExtensions.GetHelpText(this Enum)` (returns the text, or empty string if absent) — used for the longer explanatory blurb shown under a dropdown (e.g. `OverwriteBehaviour`). The namespace is imported in `Components/_Imports.razor`. (Note: in the test project NUnit's global `using NUnit.Framework` makes `Description` ambiguous with `System.ComponentModel.DescriptionAttribute` — fully-qualify it there.)
- **Blazor components use code-behind.** A `.razor` file should hold only markup and the minimal C# unavoidable in markup (e.g. simple `@bind`/event wiring); put the bulk of the logic — fields, properties, lifecycle methods, event handlers, `[Inject]` services — in a sibling `.razor.cs` partial class (`public partial class Foo { ... }` in the component's namespace). Create the code-behind whenever a component needs more than trivial inline code.
- **Each page gets its own folder under `Components/Pages/`.** A routable page lives in `Components/Pages/<PageName>/<PageName>.razor`, with its code-behind and any page-specific files (sub-components, view models, partials) in that same folder. Examples: `Components/Pages/Dashboard/`, `Components/Pages/Settings/`, `Components/Pages/Authentication/` (the Login page). The folder becomes part of the component namespace (e.g. `BackupService.Components.Pages.Authentication`); match the code-behind's `namespace` to it. Routes come from `@page`, not the folder, so moving a page between folders doesn't change its URL. (Note: the page folder `Components/Pages/Authentication/` is distinct from the root `Authentication/` folder, which holds the auth *services*.)
- **For logging, use `Microsoft.Extensions.Logging`** — inject `ILogger<T>` via the constructor (already configured by the host; no third-party logging framework like Serilog or NLog). Use structured-logging message templates (e.g. `logger.LogInformation("Backed up {File}", path)`).
- **Use dependency injection throughout, via the built-in `Microsoft.Extensions.DependencyInjection` container** (available as `builder.Services` — no third-party container like Autofac). Register services on `builder.Services` (`AddSingleton`/`AddScoped`/`AddTransient`) and inject them via constructors rather than newing them up or using static/singleton access. Backup logic should live in injectable services (interface + implementation) consumed by `Worker`, not inline in `Worker` itself.
